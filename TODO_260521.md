MakeSlide 程式改進分析報告
以下按嚴重程度分類，共識別出 13 個可改進項目：

🔴 高優先（架構/安全問題）
1. sync.ts:44 — Session 記憶體洩漏且重啟後狀態全部遺失

sessions Map 永遠不會清除已過期的 session：

每個曾使用過的 PDF 都會在記憶體中永久保留 session 物件（含問題列表、AI 回答等）
伺服器重啟後課堂同步狀態（當前頁碼、學生問題、AI 回答）完全消失
建議：加入定時 GC（每 N 分鐘掃描並刪除長期無活動的 session），或比照 regenerate_jobs 的做法將 session 狀態持久化至 SQLite

2. sync.ts:160 — client_id 身份認證強度不足

master 身份僅靠「第一個加入者」決定，且 client_id 完全由客戶端自行提供（任意字串）。任何知道 client_id 格式的人都可偽裝為 master 並控制課堂投影片翻頁。

建議：join 時後端核發不可偽造的 session token（HMAC 簽章或短效 JWT），後續所有敏感操作需帶 token 驗證，而非靠 client_id 字串作為唯一憑據

3. server.ts:47 — CORS 在所有環境都允許任何來源

await app.register(cors, { origin: true, ... });

origin: true 讓任何網站都能跨域呼叫 API，正式環境應限制為允許的來源清單。

🟡 中優先（可靠性/效能）
4. openai.ts:14 — LLM 日誌路徑依賴 process.cwd()

const LLM_REQUEST_LOG_FILE = path.join(process.cwd(), 'backend', 'data', 'llm-requests.log.jsonl');

若從不同目錄啟動伺服器（如 Docker 容器的 /app），日誌會寫入錯誤路徑。應改用 config 中的 dbPath 同目錄或專屬 logPath 設定。

5. db.ts:31 — DB 遷移無版本化、每次啟動執行 30+ 次 PRAGMA 查詢

目前以手動 columnExists() 逐一確認各欄位存在與否，隨欄位增加，啟動時 overhead 持續增長，且缺乏版本號管理。

建議：引入輕量的 schema_versions 遷移表記錄已執行版本，批次執行還未跑過的遷移腳本

6. sync.ts:48 — 每次請求都呼叫 db.prepare()

function ensurePdfExists(id: string): boolean {
  const row = db.prepare(`SELECT id FROM pdfs WHERE id = ?`).get(id) ...

應在模組頂層 prepare 一次：const stmtCheckPdf = db.prepare('SELECT id FROM pdfs WHERE id = ?') 以提升效能。

7. sync.ts:305 — 靜態依賴以動態 import() 引入

const fs = await import('node:fs');
const { safeJoinPdfPath } = await import('../../services/storage');

這兩個是靜態依賴，應移至檔案頂層 import，每次請求執行動態 import 表達式不合慣例。

8. PlayPage.tsx:1 — 仍有 3375 行

雖已完成部分模組化（play/ 子目錄），主檔仍超過 3375 行，包含 30+ 個 useState、音訊控制、同步教室、投票、AI 問答、重生等多個複雜功能。

建議：繼續提取為 custom hooks（usePlaybackSync, useAudioPlayer, useSyncQuestions, useQuiz）並拆分子元件

9. config.ts:61 — GERMINI_API_KEY 拼寫錯誤仍公開存在

GERMINI_API_KEY: z.string().optional().default(''),  // 拼寫錯誤
GEMINI_API_KEY: z.string().optional().default(''),   // 正確

兩個欄位並存造成混淆，應只保留正確拼寫並在 .env.example 加上向後相容說明後移除錯誤版本。

🟢 低優先（開發體驗/測試覆蓋）
10. backend/test/ — sync 路由完全缺乏測試

sync.ts 含有複雜的 master/follower 狀態機、問題提交、AI 回答觸發邏輯，但 backend/test/ 目錄完全沒有對應測試。

11. 前端元件測試覆蓋率不足

前端只有 api.error-mapping.test.ts 和 api.regenerate-pageops.test.ts 兩個測試，缺少對 PlayPage、HomePage、自訂 hook 的測試。

12. db.ts:9 — 注釋掉的死程式碼

//fs.mkdirSync(path.dirname(config.dbPath), { recursive: true });

此初始化邏輯已移至 server.ts 的 ensureWorkspaceRuntimePaths()，注釋行應刪除以避免閱讀時造成混淆。

13. 輪詢機制（Polling）效率議題

前端以固定 POLL_INTERVAL_MS = 3000 輪詢課堂同步狀態，規模增大（多個教室同時使用）時並發輪詢對後端 SQLite 形成壓力，翻頁同步也有最多 3 秒延遲。

建議：評估以 Server-Sent Events (SSE) 取代同步狀態輪詢，可同時降低延遲和伺服器負載
